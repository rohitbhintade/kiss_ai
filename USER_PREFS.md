- When user asks about code, provide detailed analysis referencing the actual source.
- When simplifying code, focus on eliminating real duplication across files, not just cosmetic changes. Verify with tests before and after.
- When removing redundancies in a module, first grep for structural-assertion tests (e.g. inspect.getsource + literal string checks) and for monkeypatches via module attribute access; update those tests to reference the new code location (e.g. the method the code was inlined into, or the module-level function the monkeypatch now targets).
- When inlining a method into its single caller, update all tests that called the removed method directly: structural tests should inspect the caller's source, behavioral tests should exercise the caller with appropriate preconditions, and monkeypatches should target the module-level function that the caller now imports directly.
- When adding tests to an existing test file, follow the existing patterns and naming conventions in that file.
- Concurrent state isolation tests should cover both directions of the interleaving, not just one.
- When fixing bugs involving state flags, check whether the flag represents current state or desired-future state, and guard on the appropriate one.
- When fixing frontend event routing bugs, always check both the event-filtering side and the response-routing side for consistency.
- When a per-tab operation receives an empty or missing tab identifier, treat it as a no-op (log and return) rather than silently falling back to global state or iterating across all tabs.
- Events broadcast from background threads must capture the originating tab identifier before the thread starts and set thread-local state inside the thread so broadcasts are tagged.
- When adding frontend UI guards based on a toggle state (e.g. is_worktree), read the toggle state from the DOM element that already tracks it and route to the correct handler rather than duplicating UI.
- Pre-existing lint errors in files unrelated to the current task should not be fixed if the user previously removed the commits that fixed them; leave them as-is.
- When asked to detect races and confirm with tests but NOT fix, produce one test file per audit with both structural source-pattern assertions AND behavioral simulations that replay the flawed algorithm verbatim, so a future fix causes the test to fail deterministically.
- Prefer `assert x is not None` over `self.assertIsNotNone(x)` when a mypy/pyright type-narrowing is needed for subsequent `.group(...)` calls.
- For one-shot request/response protocols whose reply is delivered via a single EventEmitter with multiple pending listeners, correlate by a per-invocation `requestId` (not a `tabId`): the requester allocates it, the backend echoes it in the reply, and each listener filters on it. The same requestId becomes the key for related per-invocation side state (e.g. UI countdowns) so that a result stops exactly one countdown.
- Multiple concurrent async operations that mutate the same array/object must share one serialization primitive. A boolean "op-in-progress" guard only blocks user-initiated duplicates; pair it with a single Promise-chain (`_opChain`/`_enqueue`) that every mutating path (including non-user paths like save handlers) runs through so they cannot interleave during `await`s.
- Replace single-slot "pending while in progress" fields with an array/queue when three-or-more concurrent arrivals are possible; a single slot silently drops every overwrite.
- When an event emitted by a global manager needs to be routed to the originator, store the originator id on the manager itself and emit it as an argument, rather than maintaining a FIFO queue in the subscriber — FIFO assumes completion order equals open order, which fails when operations can finish out of order.
- When fixing a race, update the matching regression tests to assert the fix (not the race): keep the structural + behavioural duo but flip both to verify post-fix invariants and compare concurrent outcomes against dynamically-computed strictly-serial baselines so the tests aren't brittle to irrelevant bookkeeping details.
- When a cancellation/failure event can be fired by any of N listeners on the same emitter, gate the "fire" site (not the listener side) so only the logically-correct source can emit it, and include the correlation id so each pending listener resolves only its own cancellation.
- Before removing a UI-state broadcast because it is suspected of "preventing" another UI behaviour, trace the other behaviour's actual guard: if it is already gated by a different flag (e.g. isRunning set by an earlier status event), the broadcast is not the cause and must not be removed.
- When the user asks "if so, remove it" conditional on a hypothesis, verify the hypothesis against the source before editing; report the verification result and only remove code when the hypothesis holds.
- When making a singleton UI region (e.g. modal) per-tab, keep a shared mount slot element in the static HTML and create per-tab DOM nodes lazily inside each tab object; on tab switch, detach the previous tab's nodes and attach the new tab's nodes — this preserves unsaved UI state (partial input, focus context) without serialising/restoring values.
- When per-tab UI elements of the same kind can coexist (one per tab, held as detached nodes on tab objects), style them via CSS classes rather than ids, and replace any id-based CSS selectors accordingly.
- Any module-level "which-tab-issued-this-request" routing variable paired with a singleton UI becomes dead weight once the UI is per-tab — remove it and route answers using the tab object's own id captured in its closure.
- When concurrent requests from N tabs can arrive for the same UI, give each tab its own per-tab pending state; a single global "pending slot" silently overwrites earlier tabs' requests. Use a FIFO queue only when multiple requests from the SAME tab can stack — if the UI is blocking/modal (agent waits for the answer before issuing another), a single `pending` field suffices; do not add a queue the protocol can never fill.
- After introducing a per-tab queue as a mechanical fix for cross-tab overwrites, re-examine whether multiple entries can ever exist in one tab's queue; if the per-tab protocol is strictly one-at-a-time, simplify the queue back down to a single pending slot.
- When asked to find obsolete tests after a refactor, do not delete tests by heuristic; run the full impacted test suite, separate environmental failures (e.g. playwright install / missing binaries) from genuinely obsolete failures, and only remove tests whose asserted invariants contradict the current source (e.g. assertions on removed module-global variables or removed wire messages). Preserve tests that fail only because an external tool is unavailable in the CI environment.
- When a refactor introduces a superseding test file, the new file's docstring often names the superseded tests by location — trust the docstring as a lead but verify each referenced test still exists and still contradicts current source before removing it.
- When auditing a module for bugs, read every source file before forming hypotheses, then confirm each bug with both a structural source assertion (inspect.getsource) and a behavioral integration test using real objects.
- When auditing for bugs, also check for inconsistent locking disciplines (e.g. a field written under a lock but read without one in the same class), TOCTOU gaps between lock acquisitions, and one-way flag setting (only True, never reset to False).
- When fixing audited bugs, update ALL audit tests that assert the old buggy behavior to assert the new fixed behavior, not just the tests in the current audit round.
- When a "close" or "cleanup" operation guards on state flags, also check whether an active thread exists (task_thread.is_alive) to avoid racing with threads that have started but not yet set their state flags.
- When fixing audited bugs, update the corresponding audit tests from asserting-the-bug to asserting-the-fix so the test suite validates the corrected behavior going forward.
- When standardizing sentinel values across methods (e.g. tab_id defaults), also update callers that pass the old sentinel (e.g. cmd.get returning None) to pass the new one (e.g. cmd.get with a default).
- When a webview event handler creates or destroys a shared UI element (worktree bar, autocommit bar, merge toolbar), it must first filter the event by tabId and only touch the DOM when the event targets the active tab; for background tabs, save the state change on the tab object so restoreTab can reconstruct it later. When an end or done event fires for a background tab, clear both the state flag and the saved DOM element reference so restoreTab does not re-attach a stale element.
- When a background tab event requires user interaction (e.g. merge_started shows Accept/Reject buttons), auto-switch to that tab so the user sees the UI immediately; set the tab state first, then call switchToTab so restoreTab can reconstruct the UI from the saved state.
- When a UI bar's button click handler posts a message that references a tab, capture the owning tabId at bar-creation time in a closure rather than reading activeTabId at click time, because the user may switch tabs between bar creation and button click. All bar types (worktree, autocommit, merge toolbar) must follow this pattern consistently.
- When a message handler has a default/catch-all case that silently drops events for non-active tabs, check whether those events build incremental UI state (panels, DOM fragments); if so, process them into the background tab's saved state so the UI is correct when the user switches back.
- When processing events for a background tab that share code with active-tab processing (e.g. handleOutputEvent), save and restore any global state or DOM elements that the shared code mutates as a side effect, to avoid polluting the active tab's display.
- When adding a new action button to the webview input footer, place it in the correct div (model-picker for model/file/toggle buttons, input-actions for action/send/stop buttons), add CSS styling consistent with sibling buttons, wire its disabled state in setRunningState, and test the HTML position, CSS, JS wiring, and backend dispatch.
- When any tab-switching function (createNewTab, switchToTab, closeTab) transitions to a non-running tab, it must stop the timer interval and reset t0 to null; otherwise the stale interval fires with t0=null producing Date.now()-0 (epoch milliseconds) as the elapsed time.
- When a lifecycle transition opens auxiliary UI (diff/merge views, preview tabs), snapshot the editors state before and restore it after the auxiliary UI is dismissed, so that the user returns to the same editor layout they had before. The snapshot must be per-tab (keyed by tabId in a Map) so concurrent merge flows from different tabs do not interfere.
- When an async editor-state operation (e.g. closing tabs after merge) is triggered by an event, never fire-and-forget it with void; chain it through a serialization promise so that subsequent operations (snapshots, new merges) wait for it to complete before reading or mutating editor state.
- UI features that are read-only or navigational (e.g. adjacent task scrolling) should not be gated on isRunning; only mutation operations (send, submit, model changes) should be blocked while a task runs.
- When replaying events into an auxiliary container (e.g. adjacent-task), save AND restore ALL header metrics (tokens, budget, steps) around the replay, and capture the replayed metrics on the container so scroll-based visibility switching can update the header from them.
- When a benchmark or evaluation runner invokes tools as subprocesses, build the package from local source before launching the subprocess and pass the pre-built artifact path via an environment variable so the subprocess can skip redundant rebuilds.
- Never pipe curl into sh for install scripts; download to a file first, then run it, so curl failures are caught by the shell AND chain. Add a post-install verification step to confirm the binary exists.
- When a protocol sends structured data that the receiver writes verbatim (e.g. crontab entries), have the sender supply the exact final string rather than decomposing it into fields that the receiver must reassemble; this eliminates parsing on both sides and keeps the wire format faithful to the target format.
- When adding new webview features, implement them in separate JS files and expose a minimal API from main.js via a window global object, keeping changes to main.js minimal.
- When replaying events in demo mode, group events into logical panels and replay each panel as a unit with a brief pause and collapse, rather than replaying events one-by-one with individual delays.
- When adding related work or citations to a LaTeX paper, validate every citation by checking that the arXiv ID returns HTTP 200 or the URL resolves before inserting it into the bibliography. Compile twice with pdflatex and verify no undefined citation warnings remain.
- When asked to cite recent papers, read the abstracts to judge relevance and verify publication dates before adding to the paper. Prioritize papers with high citation counts and direct topical relevance.
- Before making changes to a file, verify the current state matches the assumed precondition by reading the file and checking git history; if the file already implements what was requested (e.g. due to a prior refactor that was not reverted), report that fact instead of making redundant changes.
- When a singleton resource (e.g. editor, decoration manager) is shared across concurrent tabs, never queue or serialize multiple tabs' operations through it — instead track ownership with a single active-owner variable and defer other tabs' data in a Map, starting the next one only when the current owner finishes.
